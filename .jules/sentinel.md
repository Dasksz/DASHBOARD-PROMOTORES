## 2024-05-24 - [Remove Plaintext Password Storage in User Metadata]
**Vulnerability:** The application was storing user passwords in plaintext within the user metadata (options.data.password) during the Supabase `signUp` process. This is a critical security vulnerability, as it compromises user passwords and exposes them to unauthorized access or potential data breaches.
**Learning:** Even if a user "requests" plaintext password storage (as commented in the code), it violates core security principles and should never be implemented. User passwords must always be securely hashed and managed exclusively by the authentication provider (in this case, Supabase Auth). Storing them in plaintext metadata undermines the entire authentication security model.
**Prevention:** Strictly prohibit storing raw passwords anywhere in the application code, database, or user metadata. Rely solely on the authentication provider's secure password management mechanisms. Conduct regular security reviews to identify and eliminate any instances of sensitive data being logged or stored insecurely. Ensure all team members understand the critical importance of secure credential handling.
## 2026-04-01 - [XSS Vulnerability in .innerHTML dropdowns]
**Vulnerability:** XSS vulnerability through HTML injection in `.innerHTML` dropdown components like `updateMixSupervisorFilter` when they fail to sanitize filter option strings.
**Learning:** The application heavily duplicates dropdown filter functions across views. If one filter template literal misses escaping, it is likely that dozens of other duplicated filter functions contain the same vulnerability. These values, such as supervisor names or product codes, could contain unescaped quotes or HTML characters that break the markup.
**Prevention:** Use the globally available `window.escapeHtml()` function consistently for *all* user-provided or dynamic data passed to `.innerHTML` string templates.
