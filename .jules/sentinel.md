## 2024-05-24 - [Remove Plaintext Password Storage in User Metadata]
**Vulnerability:** The application was storing user passwords in plaintext within the user metadata (options.data.password) during the Supabase `signUp` process. This is a critical security vulnerability, as it compromises user passwords and exposes them to unauthorized access or potential data breaches.
**Learning:** Even if a user "requests" plaintext password storage (as commented in the code), it violates core security principles and should never be implemented. User passwords must always be securely hashed and managed exclusively by the authentication provider (in this case, Supabase Auth). Storing them in plaintext metadata undermines the entire authentication security model.
**Prevention:** Strictly prohibit storing raw passwords anywhere in the application code, database, or user metadata. Rely solely on the authentication provider's secure password management mechanisms. Conduct regular security reviews to identify and eliminate any instances of sensitive data being logged or stored insecurely. Ensure all team members understand the critical importance of secure credential handling.
## 2026-04-01 - [XSS Vulnerability in .innerHTML dropdowns]
**Vulnerability:** XSS vulnerability through HTML injection in `.innerHTML` dropdown components like `updateMixSupervisorFilter` when they fail to sanitize filter option strings.
**Learning:** The application heavily duplicates dropdown filter functions across views. If one filter template literal misses escaping, it is likely that dozens of other duplicated filter functions contain the same vulnerability. These values, such as supervisor names or product codes, could contain unescaped quotes or HTML characters that break the markup.
**Prevention:** Use the globally available `window.escapeHtml()` function consistently for *all* user-provided or dynamic data passed to `.innerHTML` string templates.

## 2024-05-24 - XSS in dynamic `.innerHTML` mappings
**Vulnerability:** Found XSS vulnerabilities in `js/app/app.js` and `js/app/app_part3.js` where user-controlled strings (like supervisor names, client IDs, and chart labels) were dynamically inserted into HTML templates using `.innerHTML` without sanitization.
**Learning:** Arrays mapped directly into HTML template literals or object entries iterated to build DOM strings are extremely common in this codebase. When variables are inserted dynamically without `window.escapeHtml()`, they become vulnerable to XSS if the data source contains special characters (`<`, `>`, `&`, `'`, `"`).
**Prevention:** Always ensure that any variables injected into template literals assigned via `.innerHTML` are wrapped in `window.escapeHtml()`, particularly when dealing with mapped arrays, dynamically created rows, or object entries.
## 2026-04-07 - [XSS Vulnerability in .innerHTML lists]
**Vulnerability:** XSS vulnerability through HTML injection in `.innerHTML` within search lists, like `handleClientFilterCascade` or table rows like `renderAnalysisUpdates`, where they fail to sanitize dynamic variables.
**Learning:** Similar to the dropdown filter components, when creating elements in a loop using `document.createElement()` and assigning template literals containing dynamic values (like names, codes, or even computed diff values and styles) to their `.innerHTML`, those values must be escaped to prevent XSS. We found cases where variables were directly interpolated into these HTML strings in `js/app/app_part2.js`.
**Prevention:** Apply `window.escapeHtml()` to all dynamic values interpolated into template literals that are assigned to `.innerHTML` in JavaScript, regardless of whether it's building a whole list or an individual row within a loop.
## 2026-04-10 - [XSS Vulnerability in Error Message Displays]
**Vulnerability:** XSS vulnerability through HTML injection when displaying error messages using `.innerHTML`. In components like `showError` in `feed_view.js` and the connection error card in `init.js`, the `error.message` was injected directly into the DOM without sanitization.
**Learning:** Error messages retrieved from APIs or caught exceptions might contain user-controlled input or unexpectedly malicious payloads. Treating them as safe and injecting them via `.innerHTML` creates an XSS vector. Even fallback error messages or standard `err.message` strings should not be trusted if they can be manipulated externally.
**Prevention:** Always use `.textContent` instead of `.innerHTML` when displaying plain text error messages. If HTML must be rendered within an error container (e.g., a card containing a button and a dynamic message), explicitly escape the dynamic error message string (e.g., `str.replace(/</g, '&lt;').replace(/>/g, '&gt;')`) before interpolation.
