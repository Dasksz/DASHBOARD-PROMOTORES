## 2024-05-18 : (Repeated Filter Rendering Logic) **Learning:** Multiple parts of the UI duplicate the logic to resolve valid vendors based on a selected set of supervisors, render a dropdown, and update filter text. Extracting this common logic into a utility reduces line count and centralizes DOM manipulations that need XSS protection (like `window.escapeHtml`). **Action:** Always look for and use the new generic helper `updateGenericVendedorFilter` instead of duplicating loop-based filter generations.
## 2024/05/18 : (Supervisor Filter Abstraction) **Aprendizado:** When abstracting duplicated UI generation logic (like `update[View]SupervisorFilter` across 15 views), you cannot assume the filtering logic is completely identical across the codebase. Different application modules (like "Goals") rely on subtle business logic variations (e.g. checking `d.supervisor !== '0'`). **Ação:** Always build utility functions (like `window.updateGenericSupervisorFilter`) to accept an optional `filterFn` callback, defaulting to the most common case, to allow specific views to safely inject their required logic without causing regressions. Also, ensure utility callbacks handling element IDs can gracefully accept either the string ID or the actual DOM Node if legacy functions differ.
## 2024/05/18 : (Repeated Filter Rendering Logic - Rede & TipoVenda) **Aprendizado:** UI modules extensively duplicated filter rendering logic for 'Rede' and 'TipoVenda', iterating directly via Proxy `.get()` causing GC pressure in ColumnarDatasets. **Ação:** Refactored into reusable globally exposed logic `window.updateGenericRedeFilter` and `window.updateGenericTipoVendaFilter` that implements direct columnar checks (`typeof dataSource.get === 'function' && dataSource._data`) to bypass Proxy allocation overhead, avoiding repetitive mapping implementations across different components.
## 2026/04/07 : (Remove Duplicate Files) **Aprendizado:** Unreferenced code duplicate files (like app_part1.js, mix_view_code.js, coverage_code.js, rpc_view.js) can cause confusion and bloat the workspace. These files are not imported via index.html or init.js. **Ação:** Remove such unused files carefully after confirming they aren't loaded or referenced.
## 2024-06-03 : (Main Vendedor and Supervisor Filter Refactoring) **Aprendizado:** The main `vendedor` and `supervisor` filter dropdown logic was heavily duplicated, despite having standard utility functions like `window.updateGenericSupervisorFilter`. However, `updateVendedorFilterDropdown` specifically incorporates 'Promoter Mode' logic using `getHierarchyFilteredClients`. Refactoring it entirely to the generic filter causes regressions. **Ação:** Refactored `updateSupervisorFilterDropdown` to use `window.updateGenericSupervisorFilter` but kept `updateVendedorFilterDropdown` standalone with inline improvements, explicitly commenting why it can't be merged to prevent future incorrect refactors.
## 2026/04/10 : (User Role Context Logic Extraction) **Aprendizado:** The complex logic to check a user's exact hierarchy context and role permissions (isPromoter, isManager, etc.) was duplicated across `js/app/feed_view.js` and `js/app/app.js`, making role management brittle. Directly replacing strict behavior flags like `window.userIsPromoter` with the broader hierarchy check is dangerous and breaks UI visibility constraints. **Ação:** Extracted the core check into `window.getUserRoleContext` in `js/app/utils.js`. Safely replaced the verbose duplicated `isPromotorUser` assignments in `app.js` using this central utility, while strictly preserving explicit UI restriction logic that relies on specific sub-roles.
